Threat Intelligence Fundamentals - Detection Engineering

Understanding, collecting, and operationalizing threat intelligence for detection engineering

What is Threat Intelligence?

Threat Intelligence (TI) is evidence-based knowledge about existing or emerging threats that helps inform security decisions. It transforms raw data into actionable insights that drive detection, prevention, and response capabilities.

ğŸ’¡ Key Distinction: Threat intelligence is not just a feed of IOCs - it's contextual information that enables understanding of adversary capabilities, intent, and behavior patterns.

The Intelligence Cycle

Phase Description Detection Engineering Role
Planning & Direction Define intelligence requirements and priorities Identify gaps in detection coverage, prioritize threat actors
Collection Gather raw data from various sources Ingest IOCs, threat feeds, reports, telemetry
Processing Normalize and structure collected data Parse feeds, deduplicate, standardize formats (STIX/TAXII)
Analysis Transform data into actionable intelligence Correlate with environment, assess relevance, extract TTPs
Dissemination Deliver intelligence to stakeholders Create detections, update blocklists, brief SOC teams
Feedback Evaluate effectiveness and refine requirements Measure detection hits, tune rules, improve coverage

Types of Threat Intelligence

Strategic Intelligence

High-level information for executives and decision-makers about threat landscape trends, geopolitical factors, and risk assessments.

  • Threat landscape reports
  • Industry-specific threat assessments
  • Geopolitical cyber threat analysis
  • Risk posture evaluations

Tactical Intelligence

Detailed information about adversary TTPs (Tactics, Techniques, and Procedures) used by security teams to improve defenses.

  • MITRE ATT&CK mappings
  • Attack playbooks and kill chains
  • Malware analysis reports
  • Detection rule development guidance

Operational Intelligence

Information about specific incoming attacks or campaigns enabling proactive defense.

  • Campaign tracking and attribution
  • Threat actor profiles and targeting
  • Vulnerability exploitation timelines
  • Attack infrastructure mapping

Technical Intelligence

Machine-readable indicators that can be directly operationalized in security tools.

  • IP addresses, domains, URLs
  • File hashes (MD5, SHA1, SHA256)
  • Email indicators (sender, subject, attachments)
  • Network signatures (YARA, Snort, Sigma)

Sentinel TI Integration

Built-in threat intelligence via Microsoft Defender Threat Intelligence (MDTI), threat indicator tables, and TI workbooks for visualization.

Splunk TI Framework

Enterprise Security threat intelligence framework with KV stores, threat_activity datamodel, and integration with various TI platforms.

Recommended TTL by Indicator Type

Indicator Type Recommended TTL Rationale
File Hash (MD5/SHA) 180-365 days Static, but malware becomes irrelevant
Domain 30-90 days Can change ownership, be sinkholed
IP Address 7-30 days Highly dynamic, cloud/VPS rotation
URL 7-14 days Very short-lived, campaigns end quickly
Email Address 30-90 days Campaign-specific, reuse is limited
TTP/Behavior 365+ days Most durable, adversaries slow to change

KQL: IOC Age Analysis

KQL - Analyze IOC Age Distribution 
ThreatIntelligenceIndicator
| where TimeGenerated > ago(90d)
| where Active == true
| extend IndicatorAge = datetime_diff('day', now(), CreatedTimeUtc)
| extend AgeCategory = case(
    IndicatorAge <= 7, "Fresh (0-7 days)",
    IndicatorAge <= 30, "Recent (8-30 days)",
    IndicatorAge <= 90, "Aging (31-90 days)",
    IndicatorAge <= 180, "Stale (91-180 days)",
    "Expired (180+ days)"
)
| summarize 
    IndicatorCount = count(),
    AvgConfidence = avg(ConfidenceScore)
    by AgeCategory, ThreatType
| order by AgeCategory asc

KQL - Threat Feed Quality Analysis 
// Analyze quality metrics by threat intelligence source
let TIMatches = SecurityAlert
| where TimeGenerated > ago(30d)
| where AlertType contains "TI"
| extend SourceFeed = extract(@"Source:\s*(\w+)", 1, ExtendedProperties)
| summarize 
    Alerts = count(),
    TruePositives = countif(Classification == "TruePositive"),
    FalsePositives = countif(Classification == "FalsePositive")
    by SourceFeed;
//
let TIIndicators = ThreatIntelligenceIndicator
| where TimeGenerated > ago(30d)
| summarize TotalIndicators = count() by SourceSystem;
//
TIMatches
| join kind=leftouter TIIndicators on $left.SourceFeed == $right.SourceSystem
| extend 
    TPRate = round(100.0 * TruePositives / Alerts, 2),
    FPRate = round(100.0 * FalsePositives / Alerts, 2),
    HitRate = round(100.0 * Alerts / TotalIndicators, 4)
| extend QualityScore = (TPRate * 0.5) + ((100 - FPRate) * 0.3) + (HitRate * 0.2)
| project SourceFeed, TotalIndicators, Alerts, TruePositives, FalsePositives, 
          TPRate, FPRate, HitRate, QualityScore
| order by QualityScore desc

Q: What is the Pyramid of Pain and how does it guide detection strategy?

A: The Pyramid of Pain ranks indicator types by how difficult they are for adversaries to change:

  • Bottom (Easy to change): Hash values - adversaries change a single byte
  • Middle: IP addresses, domains - infrastructure can be rotated
  • Top (Hard to change): TTPs - behavioral patterns require significant retooling

Detection strategy should prioritize higher levels for durable detections. While hash-based IOCs are easy to implement, they're also easy to evade. Behavioral detections based on TTPs (like detecting credential dumping techniques regardless of specific tool) provide longer-lasting value.

Q: How do you evaluate the quality of a threat intelligence feed?

A: I evaluate feeds across multiple dimensions:

  • Accuracy: True positive rate when indicators match - are alerts legitimate threats?
  • Relevance: Do indicators relate to our industry, technology stack, and threat profile?
  • Timeliness: How fresh are indicators? What's the delay from threat emergence to feed publication?
  • Coverage: Does the feed provide context (TTPs, attribution, related indicators)?
  • False Positive Rate: How many alerts are noise vs actionable?

I track metrics per feed and regularly compare quality scores to make retention decisions and optimize budget allocation.

Q: Explain the difference between strategic, tactical, operational, and technical threat intelligence.

A:

  • Strategic: High-level trends for executives - "Nation-state actors are targeting healthcare sector"
  • Tactical: TTPs for security teams - "APT29 uses OAuth token theft via phishing"
  • Operational: Specific campaign info - "Campaign X targeting financial institutions in Q1"
  • Technical: Machine-readable IOCs - "Block IP 192.0.2.100, hash abc123..."

As a detection engineer, I primarily consume tactical and technical intelligence - TTPs inform behavioral detection logic, while IOCs enable direct matching. However, strategic and operational context helps prioritize detection development.

Q: How do you handle IOC lifecycle management to prevent alert fatigue?

A: Effective IOC lifecycle management includes:

  • TTL Policies: Set expiration based on indicator type - IPs expire in 7-30 days, hashes in 180 days
  • Confidence Decay: Automatically reduce confidence scores over time
  • Validation: Periodically check if indicators are still active (domain still resolving, IP still malicious)
  • Deduplication: Merge overlapping indicators from multiple feeds
  • Relevance Filtering: Only ingest indicators relevant to our environment (e.g., skip Android malware if we're Windows-only)

I also implement feedback loops - when analysts mark alerts as false positives, those indicators get flagged for review or automatic removal.
