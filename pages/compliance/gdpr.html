GDPR & Privacy Regulations

Privacy regulation requirements for security monitoring, breach detection, and data protection in detection engineering.

GDPR Overview

The General Data Protection Regulation (GDPR) governs the protection of personal data for EU residents. Detection engineers must balance security monitoring needs with privacy requirements.

Key GDPR Articles for Detection

Article Title Detection Relevance
Art. 32 Security of Processing Requires appropriate security measures including detection
Art. 33 Breach Notification (Authority) ‚òÖ 72-hour notification - requires rapid detection
Art. 34 Breach Notification (Data Subject) Notify affected individuals of high-risk breaches
Art. 5 Principles Data minimization, purpose limitation in logs
Art. 30 Records of Processing Document logging activities as processing

Breach Detection Queries

Data Exfiltration Detection for GDPR 
// Detect potential personal data breach scenarios

// 1. Large data downloads from HR/PII databases
AzureDiagnostics
| where Category == "SQLSecurityAuditEvents"
| where statement_s has_any ("SELECT", "EXPORT")
| where affected_rows_d > 1000
| where database_name_s has_any ("HR", "Customer", "Employee", "PII")
| project TimeGenerated, client_ip_s, server_principal_name_s, 
          database_name_s, affected_rows_d
| extend AlertType = "GDPR_PotentialBreach_LargeDataAccess"

// 2. Email forwarding rules (data exfiltration)
OfficeActivity
| where Operation in ("New-InboxRule", "Set-InboxRule")
| where Parameters has_any ("ForwardTo", "ForwardAsAttachmentTo", "RedirectTo")
| project TimeGenerated, UserId, Operation, Parameters
| extend AlertType = "GDPR_PotentialBreach_EmailForwarding"

// 3. Unusual file sharing externally
OfficeActivity
| where Operation == "SharingSet"
| where TargetUserOrGroupType == "Guest"
| where Site_Url has_any ("HR", "Employee", "Personal", "Confidential")
| project TimeGenerated, UserId, Site_Url, TargetUserOrGroupName
| extend AlertType = "GDPR_PotentialBreach_ExternalSharing"

Privacy-Preserving Detection

Detection engineering must balance security monitoring with GDPR's data minimization principle.

Logging Considerations

‚úÖ Appropriate to Log

  • User IDs (pseudonymized if possible)
  • Timestamps
  • IP addresses (with justification)
  • Actions/events
  • System identifiers
  • Security events

‚ùå Avoid Logging

  • Actual personal data content
  • Password attempts (plain text)
  • Full request/response bodies
  • Health information
  • Financial account details
  • Biometric data

üí° Privacy-Preserving Techniques

  • Pseudonymization: Replace identifiers with tokens
  • Aggregation: Report on patterns, not individuals
  • Minimization: Log only what's necessary for security
  • Retention limits: Delete logs after defined period
  • Access controls: Restrict who can view logs with PII

Interview Preparation

Two main impacts:

  1. Breach detection requirement: 72-hour notification requires rapid detection capabilities
  2. Privacy constraints: Data minimization affects what can be logged

Balance: Security monitoring is a legitimate interest under GDPR, but must be proportionate and documented.

Article 32 requires "appropriate technical and organisational measures" including:

  • Ability to ensure ongoing confidentiality, integrity, availability
  • Ability to restore access after incident
  • Regular testing of effectiveness

Detection supports Art. 32 by:

  • Detecting threats before they cause breaches
  • Identifying compromises quickly to limit impact
  • Providing evidence for effectiveness testing
